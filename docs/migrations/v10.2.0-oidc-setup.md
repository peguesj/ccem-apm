# v10.2.0 OIDC Setup Guide

CCEM APM v10.2.0 adds enterprise IdP verification via the `assent` library.
Agent identity tokens can now be issued and verified by Okta, Auth0, Microsoft
Entra (Azure AD), or any OIDC-compliant provider.

## Overview

- New module: `ApmV5.Auth.OidcVerifier` — JWKS-caching OIDC JWT verifier
- Updated: `ApmV5.Auth.SessionStore.create/2` — accepts `oidc_id_token` opt
- New dependencies: `assent ~> 0.3`, `req ~> 0.5`, `jose ~> 1.11`
- Config-driven: zero behavior change when no providers are configured
- OIDC `sub` claim takes precedence over local `user_id` when both provided

## New Dependencies

```elixir
# mix.exs
{:assent, "~> 0.3"},
{:req,    "~> 0.5"},
{:jose,   "~> 1.11"}
```

Run `mix deps.get` after updating `mix.exs`.

## Configuration

### Okta

```elixir
# config/runtime.exs
config :apm_v5, :oidc_providers, %{
  okta: %{
    issuer:   "https://YOUR_DOMAIN.okta.com",
    audience: "api://apm-v5"
  }
}
```

Okta emits RS256-signed tokens. The `OidcVerifier` discovers the JWKS URI
from `https://YOUR_DOMAIN.okta.com/.well-known/openid-configuration` and
caches the public keys for 5 minutes.

### Auth0

```elixir
config :apm_v5, :oidc_providers, %{
  auth0: %{
    issuer:   "https://YOUR_TENANT.auth0.com/",
    audience: "https://api.your-app.com"
  }
}
```

Auth0 includes a trailing slash in the `iss` claim. Ensure the `issuer`
value matches exactly (with slash).

### Microsoft Entra (Azure AD)

```elixir
config :apm_v5, :oidc_providers, %{
  entra: %{
    issuer:   "https://login.microsoftonline.com/YOUR_TENANT_ID/v2.0",
    audience: "api://YOUR_CLIENT_ID"
  }
}
```

Entra v2.0 tokens use RS256. The `aud` claim matches the Application (client)
ID URI or the configured `api://` scope.

### Multiple Providers

Multiple providers can be active simultaneously:

```elixir
config :apm_v5, :oidc_providers, %{
  okta:  %{issuer: "https://corp.okta.com",                   audience: "api://apm-v5"},
  auth0: %{issuer: "https://dev-abc.auth0.com/",              audience: "https://api.apm.example.com"},
  entra: %{issuer: "https://login.microsoftonline.com/TENANT/v2.0", audience: "api://CLIENT"}
}
```

The `provider` key is passed at call-time:

```elixir
SessionStore.create(user_id, role, oidc_id_token: jwt, oidc_provider: :okta)
```

## Using OIDC in SessionStore

```elixir
alias ApmV5.Auth.SessionStore

# With OIDC verification — sub claim becomes user_id
case SessionStore.create("ignored", "agent",
       oidc_id_token: id_token_from_idp,
       oidc_provider: :okta
     ) do
  {:ok, session_id} ->
    session = SessionStore.get(session_id)
    # session.user_id == okta sub claim
    # session.metadata.oidc_verified == true
    # session.metadata.oidc_provider == :okta

  {:error, :token_expired}    -> # token TTL exceeded
  {:error, :invalid_audience} -> # aud mismatch
  {:error, :invalid_issuer}   -> # iss mismatch
  {:error, :invalid_signature}-> # JWKS verification failed
end

# Without OIDC — identical to v9.3.0
{:ok, session_id} = SessionStore.create("agent-local-001", "agent")
```

## Direct OidcVerifier Usage

```elixir
alias ApmV5.Auth.OidcVerifier

# Named GenServer (started by Application supervisor when providers configured)
case OidcVerifier.verify_id_token(id_token, :okta) do
  {:ok, claims} ->
    claims["sub"]   # authoritative identity

  {:error, :no_oidc_providers_configured} ->
    # No providers in Application env — OIDC disabled

  {:error, :provider_not_configured} ->
    # :okta not in the providers map

  {:error, reason} ->
    # :token_expired | :invalid_audience | :invalid_issuer | :invalid_signature
end
```

## Application Supervisor

When `oidc_providers` is configured (non-empty map), add `OidcVerifier` to
your supervision tree in `application.ex`:

```elixir
# lib/apm_v5/application.ex
children = [
  # ... existing children ...
  {ApmV5.Auth.OidcVerifier, []}   # starts only when providers configured
]
```

`OidcVerifier.start_link/1` reads `Application.get_env(:apm_v5, :oidc_providers, %{})`
on startup. If the map is empty the process still starts but all verify calls
return `{:error, :no_oidc_providers_configured}`.

## JWKS Cache

| Setting       | Value    | Override                            |
|---------------|----------|-------------------------------------|
| TTL           | 300 s    | N/A — compile constant for now      |
| Storage       | GenServer state (per-provider map) | — |
| On expiry     | Lazy re-fetch on next verify call  | — |
| On JWKS error | Empty key set → `:invalid_signature` | — |

## Backward Compatibility

- If `config :apm_v5, :oidc_providers` is absent or `%{}`, all behavior is
  identical to v9.3.0. No changes to existing `SessionStore.create/2` callers.
- The `oidc_id_token` option is ignored (not present in earlier call sites).
- `OidcVerifier.verify_id_token/2` returns `{:error, :no_oidc_providers_configured}`
  when `OidcVerifier` is not running.

## v10.0.0 JWT Assertion Composition

v10.0.0 introduced RFC 7523 JWT Bearer Assertion via `SessionStore` opts.
When both `oidc_id_token` and `jwt_assertion` are provided, OIDC takes
precedence — the verified OIDC `sub` becomes `user_id`.

## Security Notes

- All token signature verification uses JWKS public keys from the IdP's
  well-known endpoint. Symmetric (HS256) keys are supported only in test mode.
- JWKS keys are never persisted to disk; they are only held in the OidcVerifier
  GenServer's memory.
- Token expiry (`exp`) and audience (`aud`) are always validated.
- Issuer (`iss`) is validated against the configured `issuer` value.
