# v10.0.0 Migration — Agent Identity JWT Bearer Assertions

**Sprint:** v10.0.0 LINCHPIN — Wave B
**Stories:** auth-v10.0-s1 (CP-289), auth-v10.0-s2 (CP-290)
**Plane:** [v10.0.0/s1](https://plane.lgtm.build/lgtm/projects/a20e1d2e-3139-406e-ae03-dc6d1d8cb995/issues/cb77876b-0d2e-4d17-9e66-615ddd7a1817/), [v10.0.0/s2](https://plane.lgtm.build/lgtm/projects/a20e1d2e-3139-406e-ae03-dc6d1d8cb995/issues/3f371061-034c-42a9-ac7d-3bf37fa38a9f/)
**DRTW reference:** `docs/drtw-governance/01-authorization.md` GAP 1 (JWT) + GAP 2 (Delegation)

> **BREAKING (with 6-month deprecation window).** v10.0 introduces cryptographic verification of agent identity. Callers that do not migrate within the deprecation window will be rejected once enforcement lands.

---

## Background

Before v10.0.0, `agent_id` was a plain string passed in hook/API payloads and trusted at face value. There was zero cryptographic binding between an `agent_id` and the actual agent process that asserted it. Any caller could spoof any other agent's identity — including escalating privileged sub-agents.

This was the **linchpin gap** in the entire authorization stack. Every downstream control (policy decisions, rate limits, audit trails, delegation chains) ultimately trusted whatever `agent_id` arrived in the request body.

v10.0.0 fixes this via two complementary mechanisms:

1. **JWT Bearer Assertions** (RFC 7523) — Ed25519-signed agent identity tokens.
2. **DelegationToken** — Ed25519-signed parent→child capability tokens that enforce OWASP MCP02 (no scope creep).

## Story 1 — RFC 7523 JWT Bearer Assertions (CP-289)

### What changed

- New module: `ApmV5.Auth.JwtAssertion`
  - `sign_assertion(claims, opts)` — Ed25519-signed JWT (header.payload.signature, base64url).
  - `verify_assertion(jwt, opts)` — verifies signature + expiration. Returns `{:ok, claims}` or `{:error, reason}`.
  - Algorithm: **EdDSA / Ed25519** only. No other algs supported — refusing them prevents alg-substitution attacks.
  - Pure OTP `:crypto`; **no joken dep added**. Consistent with v9.4.0 KeyStore discipline.

- `AuthorizationGate.authorize/5` now accepts optional `:identity_token` (or `"identity_token"` string-key) in `params`.
  - When present and valid: `claims["agent_id"]` (cryptographically verified) replaces the payload-supplied `agent_id`. `:agent_id_verified` and `:jwt_claims` are added to params for downstream audit.
  - When present and invalid: authorization is **denied** with `{:error, :invalid_token, ...}`. We never silently fall through when a Bearer token is asserted.
  - When absent: legacy behavior. `:agent_id_unverified` is set in params for deprecation tracking.

- `POST /api/register` now returns `identity_token` in the response body — an Ed25519-signed JWT that the caller should retain and attach as `Authorization: Bearer <jwt>` (or pass as `identity_token` in subsequent request bodies).

### Token format

```
<base64url(header)>.<base64url(payload)>.<base64url(signature)>
```

Header: `{"alg":"EdDSA","typ":"JWT"}`

Standard claims added at sign time:
- `iat` — issued-at (UNIX seconds)
- `exp` — expires-at (`iat + ttl_seconds`, default 3600)
- `jti` — unique JWT ID (UUID v4)

Caller-supplied claims (always included):
- `agent_id` (required)
- `formation_id`, `invoked_by`, `parent_agent_id`, `session_id` (optional)

### Migration steps (for hook authors and API clients)

1. **Pre-deprecation phase (now)**: Read `identity_token` from `POST /api/register` response. Store it associated with the agent's session.
2. **Adoption phase**: Attach the token on every subsequent call:
    - HTTP: `Authorization: Bearer <jwt>`
    - In-process: pass `:identity_token` in `params` to `AuthorizationGate.authorize/5`.
3. **Enforcement phase (v10.1)**: Legacy callers (no token) will receive `{:error, :token_required}`.

### Test coverage

`test/apm_v5/auth/jwt_assertion_test.exs` (10 tests):
- sign + verify roundtrip
- tampered payload fails (byte-flip in payload, signature over original)
- tampered signature fails (raw byte XOR, base64 re-encode)
- expired JWT rejected (ttl=-1)
- wrong-issuer JWT rejected (signed by alt KeyStore)
- jti is unique across two signs of identical claims
- header advertises `alg=EdDSA`, `typ=JWT`
- malformed token returns `:malformed_token` (covers `""`, `"only.two"`, `"not-a-jwt"`)
- ttl defaults to 3600
- custom ttl honored

`test/apm_v5/auth/authorization_gate_jwt_test.exs` (6 tests):
- rejects malformed Bearer token gracefully
- rejects expired Bearer token
- rejects tampered Bearer token
- accepts valid Bearer + verifies agent_id from claims
- absent Bearer → legacy path, no crash
- string-keyed `identity_token` accepted (HTTP path)

### Backward compatibility

- API consumers ignoring the new `identity_token` response field: unaffected (additive).
- `AuthorizationGate.authorize/5` callers passing no `:identity_token`: legacy path with `:agent_id_unverified=true` flag for audit. **No behavior change** in this release.
- Hook clients: should adopt before v10.1.

## Story 2 — DelegationToken (OWASP MCP02 fix) (CP-290)

### Problem

Currently a parent agent can register a child agent with permissions exceeding its own ceiling. This is OWASP MCP02 (Tool / Capability Scope Creep): privileged sub-agents that escape parent constraints.

### What changed

- New module: `ApmV5.Auth.DelegationToken`
  - Struct: `{parent_agent_id, child_agent_id, max_risk_ceiling, allowed_tools, expires_at, signature}`
  - `issue(parent_token | nil, child_agent_id, opts)` — issues a child token. If `parent_token` is supplied:
    - Child's `allowed_tools` MUST be `⊆` parent's `allowed_tools` (set inclusion).
    - Child's `max_risk_ceiling` MUST be `≤` parent's `max_risk_ceiling` (ordered `:none < :low < :medium < :high < :critical`).
    - Otherwise returns `{:error, :exceeds_parent_ceiling}`.
  - `verify(token)` — Ed25519 signature check + expiration check.
  - `enforce_ceiling(token, requested_tool, requested_risk_level)` — gates an individual tool call:
    - `{:error, :tool_not_allowed}` if tool not in `allowed_tools`.
    - `{:error, :risk_exceeds_ceiling}` if risk level exceeds `max_risk_ceiling`.

### Integration

When a parent registers a sub-agent (via formation orchestration or direct call), the parent's `DelegationToken` (or its JWT assertion's chain claim) is passed. The child's token is issued via `DelegationToken.issue(parent_token, child_agent_id, ...)` — **the child's ceiling is mathematically guaranteed `⊆` parent's**.

### Test coverage

`test/apm_v5/auth/delegation_token_test.exs` (8+ tests):
- child token with 3 tools when parent has 5 → ok (subset)
- child token with 1 extra tool not in parent → reject
- child risk_ceiling higher than parent → reject
- expired token rejected
- enforce_ceiling correctly gates allowed tool calls
- enforce_ceiling rejects disallowed tool calls
- root token (no parent) can be issued with any scope
- tampered signature fails verification

### Note on DelegationChain (v9.4.0 prov-w3)

`ralph/v9.4.0-prov-w3` is on the same commit as `identity-foundation` — `DelegationChain` has NOT yet landed at the time of writing this story. `DelegationToken` is therefore implemented as a standalone primitive that exports its `parent_agent_id` chain pointer. When `DelegationChain` ships, `DelegationToken.issue/3` will gain an `:include_in_chain` option to publish the issuance event into the chain.

### Migration steps

1. Formation orchestrators that spawn sub-agents: when registering a child, call `DelegationToken.issue(my_token, child_id, allowed_tools: subset_of_my_tools, max_risk_ceiling: <= mine)`.
2. Hook handlers receiving sub-agent tool calls: call `DelegationToken.enforce_ceiling(token, tool, risk)` before forwarding to `AuthorizationGate`.

## Combined effect

After v10.0.0, the answer to "is this agent_id really this agent?" is **cryptographically yes/no** (JWT). The answer to "is this child allowed to do what it's asking?" is **mathematically bounded by parent** (DelegationToken).

These two primitives together close the linchpin gap that GAP 1 and GAP 2 of `docs/drtw-governance/01-authorization.md` identified.
